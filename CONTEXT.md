## 🎯 Назначение

**CourtHarvest2** — CLI-инструмент для сбора и поддержания эталонного справочника судов РФ через DaData API (suggestions.dadata.ru).

В отличие от v1, которая собиралась итеративно (9 фаз, алфавитный перебор, эвристики), v2 строится на точном знании структуры кодового пространства и гарантированном переборе.

---

## 📐 Ключевые принципы

### 1. Кодовое пространство конечно и известно

Все коды судов имеют формат `RRTTNNNN`:
- `RR` = регион (**00–99**, 100 регионов)
- `TT` = тип суда (14 вариантов: RS, MS, OS, AS, GV, OV, KV, AV, KJ, AJ, AA, AO, VS, AI)
- `NNNN` = номер (0000–9999)

**1 400** возможных комбинаций RRTT (100 × 14). Из них ~478 существуют, остальные гарантированно пусты.

> **История:** В первоначальной версии было 99 регионов (01–99). Регион 00 был пропущен — из-за этого Верховный Суд (`00VS0000`) не попадал в инвентаризацию. Исправлено в коммите `4fb1f00`.

### 2. Полнота через простой перебор, не через умные эвристики

Вместо алфавитного поиска по названиям с maxDepth (v1) — **прямой перебор кодов блоками**.

Блок `RRTTAB` (2 цифры номера) = до 100 кодов. DaData возвращает ≤ 20 результатов. 100 кодов с ≤ 20 существующими судами — безопасно для всех типов, кроме самых плотных (MS в Москве). В горячих блоках (ровно 20) — углубление до 3 цифр.

### 3. MS и RS — единственные «тяжёлые» типы

Анализ собранной базы v1:

| Тип | Суда | Префиксов ~ | MAX (типичный) |
|-----|:----:|:-----------:|:--------------:|
| MS | 5 526 | 80 | 10–150 |
| RS | 1 937 | 85 | 5–50 |
| OS | 68 | 68 | 0 (один на регион) |
| AS | 81 | 81 | ~1 |
| Остальные | ~300 | ~80 | 0–5 |

OS, AS, VS, GV и прочие — **1 или несколько судов на префикс**. Их проще всего сканировать одним запросом `suggestCourt(RRTT, count=20)` вместо блочного перебора. Однако для OS/VS/AV/KV код всегда `RRTT0000` — нужно искать по точному коду, не по префиксу названия.

### 4. SuperHard — единственный режим с гарантией

Обычный refresh (tails) может пропустить суды с разреженной нумерацией (дыра v1 — OS0000, 20 пустых = стоп). SuperHard перебирает всё кодовое пространство MS/RS по 100 блокам — никаких эвристик, никаких пропусков.

**Цена:** 20 000 запросов = 2 ключа = 1 день. Раз в квартал.

## 🗂 Режимы работы

### harvest — первичный сбор

Трёхфазный процесс в одной сессии KeyManager:

```
Фаза 0: RRTT-инвентаризация     →  1 400 запросов (100 × 14)
Фаза 1: SuperHard MS/RS         → 16 000–18 000 запросов
Фаза 2: Одиночные типы          →   ~300 запросов
Фаза 3: Сборка courts.json      →   локально
                                  ─────────────────────
                             ≈ 18 000–20 000 запросов (2 ключа)
```

### superhard — тотальное сканирование

Для уже проинвентаризированной базы — перебор всех unscanned MS/RS префиксов.

### refresh — ежемесячное обновление

> **🧭 В режиме обсуждения.** Будет уточнён после реализации harvest и superhard и сверки с данными v1.

## 🧠 Known Limitations (ограничения источника DaData)

### 1. DaData не отражает ПСП и множественные адреса

DaData хранит **один адрес на один код суда**. Если у суда есть ПСП (Постоянное Судебное Присутствие) в другом населённом пункте — DaData об этом не знает. Код суда один, адрес — только основной.

**Пример:** Верещагинский районный суд Пермского края (`59RS0014`) имеет два адреса:
- Основной: г. Верещагино, ул. Фрунзе, 74
- ПСП: с. Сива, ул. Ленина, 49

DaData возвращает только первый. Второй адрес «приписан» к мировому суду `59MS0116`, хотя по факту это подразделение районного суда.

**Влияние на алгоритм:** не влияет — сбор идёт по кодам, ПСП не имеет отдельного кода. Но база не пригодна для задач, требующих привязки к территориальной подсудности без дополнительного обогащения.

### 2. DaData может отставать от реальности

Суд может физически работать, рассматривать дела, но отсутствовать в DaData. Пример:

| Код | Суд | В DaData | Реально |
|-----|-----|:--------:|:-------:|
| `42RS0022` | Ленинск-Кузнецкий районный суд Кемеровской обл. | ❌ Нет | ✅ Работает, дела назначаются |
| `31RS0010` | Ивнянский районный суд Белгородской обл. | ❌ Нет | ⚠️ Упраздняется, → ПСП |
| `31RS0021` | Старооскольский районный суд Белгородской обл. | ❌ Нет | ⚠️ Упраздняется, → ПСП |
| `81RS0001` | Косинский районный суд Пермского края | ❌ Нет | ⚠️ Упраздняется, → ПСП |

**Причина:** DaData получает данные из официальных источников (ФНС, Росстат). При реорганизации судов (слияние, переименование, объединение с городским судом) старый код может быть деактивирован в DaData раньше, чем суд фактически прекратит работу. Новый код может не появиться сразу.

**Влияние на алгоритм:** мы не можем гарантировать 100% соответствия базы реальности — только 100% соответствие тому, что отдаёт DaData на момент запроса. Это фундаментальное ограничение источника, не алгоритма.

### 3. Коды судов меняются при реорганизации

При слиянии районного суда с городским код может измениться (например, `42RS0022` → `42RS0011`). DaData может удалить старый код, но не обязательно добавляет новый немедленно. Физически здание суда и сотрудники остаются, но юридический код меняется.

## 🐛 Баги, найденные и исправленные

### Баг 1: Нет трекинга запросов в сканерах

**Симптом:** Inventory и BlockScan не сообщали KeyManager о запросах. Ключ не ротировался. При превышении 10 000 — 403.

**Фикс:** Внедрён `trackRequest`-колбэк, вызов перед каждым `suggestCourt()`.

### Баг 2: Старый ApiClient после ротации ключа

**Симптом:** scanPrefix держал ссылку на shutdown-нутый ApiClient.

**Фикс:** Фабрика `getClient: () => ApiClient` вместо готового экземпляра.

### Баг 3: Новый KeyManager для фазы 2 начинал с уже сожжённого ключа

**Симптом:** km2 инициализировался с ключа, который уже сжёг 9 500 запросов. Первый же запрос мог упасть с 403.

**Фикс:** Фаза 2 работает на том же KeyManager. try/catch на случай исчерпания.

### Баг 4: Регион 00 не учтён

**Симптом:** Верховный Суд (`00VS0000`) не находился. Итоговый счёт: 10 123 против 10 206 в v1.

**Источник:** `Array.from({ length: 99 })` — не включал регион 00.

**Фикс:** `Array.from({ length: 100 })`.

### Баг 5: ProgressTracker — визуальный шум

**Симптомы:**
- `⏱ 258386` — сырые миллисекунды вместо форматированного времени
- Дублирование строк из-за autoLog
- `📊 0 судов` всю фазу 0 (нет `addFound`)

**Фикс:** fmt elapsed, autoLog удалён, addFound добавлен.

### Баг 6: Фаза 2 искала одиночные типы по префиксу названия, а не по коду

**Симптом:** OS (68 судов), VS, AV, KV не найдены. `suggestCourt("01OS", count=20)` ищет по названию, а в названии нет «01OS».

**Фикс:** Трёхшаговый probe:
1. `suggestCourt(RRTT0000, 1)` — ловит OS, VS, AV, KV (номер 0000)
2. Если пусто: `suggestCourt(RRTT0001, 1)` — ловит AS, AA, AO, AJ, KJ (номер 0001)
3. Если пусто и тип не OS/VS: блочный поиск RRTT00–99 (для GV, OV с >1 суда)

## ⚙️ Технические решения

### Паттерн getClient / trackRequest

```
Перед каждым API-вызовом:
  1. trackRequest() — увеличить счётчик, при лимите → rotate
  2. getClient()    — получить актуальный (возможно, новый) ApiClient
  3. suggestCourt() — HTTP-запрос к DaData
```

### ProgressTracker

Центральный класс для отображения хода работ:
- `begin(name, total)` — старт фазы
- `tick()` — одна единица работы (тихо, без вывода)
- `log(msg)` — явный лог с таймстампом `[HH:MM:SS]`
- `statusLine()` — `⏱ 5м 14с · 💳 2.env (3125/9500) · 📊 1089 суд. · 📡 3190 запр.`
- `progressLine()` — как statusLine + `15/179 (8.4%) · ETA: 1ч 20м`
- `fmt(ms)` — форматирование ms в `Xч Yм Zс`
- Пробрасывает `trackRequest()` и `getClient()` в KeyManager

### Фаза 2: Стратегия сбора одиночных типов

Одиночные типы (AS, OS, GV, AA, AO, AJ, KJ, OV, KV, AV, VS) — не MS/RS.
Трёхшаговый probe для каждого префикса:

```
Шаг 1: suggestCourt(RRTT0000, 1)  — для OS, VS, AV, KV (номер 0000)
Шаг 2: suggestCourt(RRTT0001, 1)  — для AS, AA, AO, AJ, KJ (номер 0001)
Шаг 3: blockSearch(RRTT00-99)     — для GV, OV (множественные номера)
```

> **Важно:** Не использовать `suggestCourt(RRTT, 20)` — DaData suggest ищет по названию, а не по коду. `"01OS"` не найдёт «Областной суд Адыгеи».

### Сборка courts.json (фаза 3)

После всех фаз собирает `data/prefixes/*.json` в единый `data/courts.json`.
Формат — как в v1: `{ meta: { totalCourts, timestamp, phase }, courts: [...] }`.
Сортировка по `code`. Сохраняется в корне `data/`.

### Замена dotenv

Собственный загрузчик `src/env.ts` (~15 строк). Читает `.env`, парсит key=value, устанавливает в `process.env`. Нуль зависимостей.

### Module Resolution: bundler

`"moduleResolution": "bundler"` — оптимально для tsx. Без `.js`-суффиксов в импортах.

### TypeScript 7 + erasableSyntaxOnly

`erasableSyntaxOnly` запрещает декораторы, enum и **parameter properties** (`constructor(private x: T)` — писать нельзя, нужно явное поле + присваивание).

### Нет eslint

`strict: true` + `noUnusedLocals` + `noUnusedParameters` + `noImplicitReturns` в tsconfig покрывают типичные проверки. Prettier — форматирование.

### Ключи на Windows

Проект работает на Windows через git-bash. `path.join`, `dirname`, `mkdirSync` корректно обрабатывают `\`-разделители. `readFileSync` читает `.env` файлы без BOM-заголовков.
